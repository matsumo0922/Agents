# Dig

プラン・設計ドキュメント・会話中の実装方針を対象に、「書かれていないのに当然とされていること」を炙り出す反復インタビューを行う。明確化ではなく前提への挑戦が目的で、ユーザーが「それは考えていなかった」と言う質問を引き出せたら成功。実装や修正は行わない。

Codex / Claude Code など複数の実行環境から使われるため、特定製品の機能名に依存しない。「構造化質問ツール」は Claude Code では AskUserQuestion、Codex では request_user_input を指し、どちらも無い環境では番号付き選択肢のテキスト質問で代替する。環境固有の最適化は末尾の「環境別ヒント」に従う。

## 全体方針

- 深く狭く。広く浅いチェックリストを作らない。1つの糸を答えが出なくなるまで追ってから次に移る。
- 明確化ではなく挑戦。妥当に見える前提こそ疑う。「どちらでもよい詳細の確認」に質問枠を使わない。
- main agent は薄いオーケストレーターに徹する。ドキュメント読み込みと前提分析は分析サブエージェントに委任し、main agent は質問の提示・回答の記録・終了判断だけを行う。対象プランの全文や参照仕様を main agent のコンテキストに読み込まない。
- 質問はメイン会話からしか出せない。このスキル自体をサブエージェントや fork されたコンテキストで実行してはならない（構造化質問ツールがユーザーに届かない）。
- 例外: 対象が小さい場合（プランが数十行以内で参照ドキュメントが無い等）は、サブエージェントを使わずインラインで分析してよい。spawn のオーバーヘッドの方が高くつくため。

## 記録ファイル

`/tmp/dig/<slug>.md` を作成し、全フェーズの共有台帳とする。slug は対象を表す短い英語スラッグ。分析の全文はこのファイルに置き、main agent とサブエージェントの間では要約だけをやり取りする。

```markdown
# Dig: <対象の1行説明>

## 対象
- <プランの所在（パス or 会話中の方針の要約）、ステートされたゴールと制約>

## 前提マップ
| # | 前提 | 分類 | リスク | 状態 |
<分類: 実現可能性 / ユーザー / スコープ / 依存 / 期間 / アーキテクチャ。リスク: high / medium / low。状態: 未検証 / 質問済み / 決定 / 保留>

## ラウンドログ
### Round <N>
- <質問と回答の要約>

## Decisions
| 論点 | 決定 | 理由 | リスク | 未解決事項 |
```

`/tmp` は永続しない。ユーザーが決定を残したいと言った場合は、リポジトリ内のドキュメント（例: `docs/plans/`）へのコピーを提案する。勝手にリポジトリへ書き込まない。

## Phase 1-2: 分析サブエージェント

main agent は対象を特定し（引数 → 会話文脈 → 直近のプラン提案の順）、記録ファイルの雛形を作ってから、分析サブエージェントを spawn する。

```text
<analyst_instruction>
あなたは dig の分析サブエージェントです。コードやドキュメントを変更せず、分析だけを行ってください。

<scope>
- 対象: <プランのパス、または会話からの要約>
- 記録ファイル: /tmp/dig/<slug>.md（分析の全文はここに書き、main agent には要約だけ返す）
</scope>

<tasks>
1. 対象プラン、CLAUDE.md / AGENTS.md、プランが参照する仕様・設計ドキュメントを読む。
2. 「書かれていないのに当然とされていること」を、実現可能性 / ユーザー / スコープ / 依存 / 期間 / アーキテクチャの6分類で洗い出す。会話やプラン本文に既に明記された決定は前提マップに状態「決定」で登録し、質問対象から除外する。
3. 各前提を「間違っていた場合の被害の大きさ」でランク付けし、記録ファイルの前提マップに全件書き込む。
4. 最高リスクの前提から、次のラウンドで聞くべき質問案を2〜3問作る。各質問は選択肢2〜4個、選択肢ごとに trade-off の1行要約を付ける。CLAUDE.md / AGENTS.md の規約と矛盾する選択肢を作らない。
</tasks>

<rules>
- 自分のサブエージェントを spawn しないでください。
- ユーザーへの質問はできません。質問案を作るのが仕事です。
- 明確化質問ではなく、前提を覆しうる質問を優先してください。
</rules>

<deliverables>
main agent には次だけを返してください。ドキュメント本文や分析の全文は返さないでください。
1. 高リスク前提トップ3〜5（各1行）
2. 質問案2〜3問（質問文・選択肢・各選択肢の trade-off 1行）
3. 記録ファイルへの書き込み完了の確認
</deliverables>
</analyst_instruction>
```

## Phase 3: 質問ラウンド

main agent は質問案を構造化質問ツールでユーザーに提示する。上限は3ラウンド。

- 1ラウンド2〜3問に収める（Codex の request_user_input は1〜3問のため、共通で3問を上限とする）。
- 各質問は選択肢2〜4個。ラベルは短く、trade-off は説明側に書く。
- 複数選択は「どの領域を掘るか」の絞り込み用途に限る。
- 自由回答は環境側の「その他」入力で拾えるため、選択肢として用意しない。
- どの選択肢が選ばれてもプランが変わらない質問は提示せず、次の候補に差し替える。
- 質問案が的外れだと main agent が判断した場合は、提示前に文言を調整してよい。ただし新しい論点の発明はサブエージェントに任せる。

回答を得たら記録ファイルのラウンドログと Decisions に追記し、同じ分析サブエージェントを継続して次ラウンドを依頼する（再 spawn しない。継続できない環境では前回の要約と回答を添えて再依頼する）。

```text
<followup_request>
- ユーザーの回答: <質問ごとの選択と自由記述>
- タスク: 回答が新たに露出させた前提を特定し、最も深掘りに値する糸を1つ選んで次の質問案2〜3問を作る。記録ファイルの前提マップとラウンドログを更新する。
- 前提マップの high リスク前提がすべて「質問済み / 決定 / 保留」になったら、質問案の代わりに DRY とだけ返す。それまでは DRY を返さない。
</followup_request>
```

回答から掘るのが dig の本体である。Round 1 で「WebSocket を選んだ」なら、Round 2 は「ステートレスな API サーバ3台でセッション維持をどうするか」のように、回答が生んだ新しい前提を追う。

## Phase 4: 終了判定と延長

次のいずれかで質問ラウンドを終了する。

- 分析サブエージェントが DRY を返した
- 3ラウンドに到達した
- ユーザーに質問疲れの兆候がある（回答が雑になる、「その他」で先に進めたい旨が書かれる等）。この場合は早めに切り上げる

3ラウンド到達時に high リスクの前提が未検証のまま残っている場合のみ、「もう1ラウンド掘るか、残リスクとして記録して終えるか」を1問だけ聞く。延長は1回につき1ラウンド。

## 最終サマリー

終了時、記録ファイルの Decisions を確定させ、チャットには次だけを出す。全文の再掲はしない。

```markdown
## Dig サマリー

- 対象: <1行>
- ラウンド: <N> / 質問: <N> / 決定: <N>

### 最重要の発見
1. <ユーザーが考えていなかった論点トップ3>

### 残リスク
- <未解決のまま受け入れた前提。なければ「なし」>

### 推奨する次のアクション
1. <決定を受けてプランに反映すべきこと>

記録ファイル: /tmp/dig/<slug>.md
```

## 環境別ヒント

### Claude Code

- 構造化質問は AskUserQuestion。1回の呼び出しで最大4問だが、本スキルでは2〜3問に抑える。header は12文字以内。
- 分析サブエージェントは Agent tool で spawn し、初回は高めの effort を指定する。2ラウンド目以降は SendMessage で同じサブエージェントを継続し（コンテキスト維持）、差分分析なので effort を下げてよい。プラン全文と参照仕様を読む長文脈役のため、最軽量ティアのモデルには切り替えない。
- 選択肢に具体的なコード案やレイアウト案を並べて比較させる場合のみ、option の preview を使う。
- plan mode 中に呼ばれた場合は、ExitPlanMode でプランを提示する前に dig を実行し、決定をプランに反映してから提示する。
- このスキルを Agent tool 経由のサブエージェントとして実行しない（AskUserQuestion がユーザーに届かない）。

### Codex

- 構造化質問は request_user_input（1〜3問）。自由記述が欲しい選択肢には isOther を使う。
- サブエージェントの spawn が explicit-only 設定などで使えない場合はインライン分析にフォールバックし、読む対象をプラン本体と直接参照されるドキュメントだけに絞る。
- 読み込み中心の分析には組み込みの explorer エージェントが向く。役割別の model / reasoning effort は `~/.codex/agents/*.toml` の `model_reasoning_effort` で調整できる。分析は長文脈読解のため、モデルはバランスティア以上（`gpt-5.6-terra` 等）を維持し、`gpt-5.6-luna` に落とさない。
- `update_plan` はチェックリスト管理ツールであり、決定の記録先にしない。記録は必ず記録ファイルに書く。
