## 必須セクション

「## 設計」は次の 8 セクションで構成する。順序もこの通りとする。

### `### 採用アプローチ`

アプローチと採用理由。候補比較を行った場合は推し案だけを書き、比較過程は台帳に残す。

### `### 不採用案`

検討して不採用にした代替案と、不採用の理由を各 1〜2 行。なければ「なし」。

### `### 事実と仮定`

設計が依拠する事実と仮定を分離する 2 表構成。

事実表 — 実際に確認した値・挙動だけを書く。確認していないものを事実表に書かない:

| 値・挙動 | 正本の所在 | 確認方法 |
|---|---|---|
| （例: 初期資金の保存値） | config / DB 保存値 / runtime 値 / 表示値 のどれか | 確認に使ったコマンド・ファイル・クエリ |

同じ論理値が複数の所在（config と DB 保存値など）を持つ場合は行を分け、値が一致しているかを明記する。不一致はそれ自体が設計の入力になる。

仮定表 — 確認できていない前提を書く:

| 仮定 | 検証状態 | 間違っていた場合の影響 | リスク |
|---|---|---|---|
| | 検証済み / 未検証 | | high / low |

high リスクかつ未検証の仮定を残したまま設計を確定しない。処置は[高リスク未検証前提プロトコル](#高リスク未検証前提プロトコル)に従う。

### `### 変更予定ファイル`

「ファイル → 何をするか」の一覧。新規コンポーネントは置き場所（module / package）と責務を決める。

### `### エッジケースと決定事項`

既知のリスク・エッジケースと決定した扱い。各決定に[帰属](#帰属ルール)を付け、「〜とする」の形で書いて読み手に委ねない。条件付きセクションの非該当宣言もここに書く。

### `### スコープ判定`

「この設計・実装・レビューを 1 worker / 1 reviewer で網羅できるか」の判断と根拠を 1〜3 行で書く。根拠には客観 signal を記録する: 跨る layer 数 / 不可逆 migration の有無 / security boundary の変更 / consumer 数 / DB hot path の有無。signal は警告であり強制 threshold ではない。最終判断は designer が行う。

網羅不能と判断した場合は staged PR 分割案を書く: 各 stage のスコープ・依存順と、各 stage が単独で merge・deploy 可能かつ backward-compatible であること（これを満たさない分割は分割として成立しない）。元の受け入れ条件は変更せず保持し、stage mapping（各受け入れ条件 → stage 1 / 後続 stage / non-goal）を付ける。

### `### 反証`

設計確定前の反証パスの結果。実施方法は `falsifier-rubric.md` に従う。各反例に処置（設計修正済み / 受容 + 理由）を付ける。反例ゼロの場合も「各ベクトルで探索した結果反例なし」と明記する。

### `### レビュー観点`

この変更固有に確認すべき観点。reviewer round 1 の一次チェックリストになるため、観測可能な形（コマンド・URL・表示・grep）で書く。

## 条件付きセクション

発動条件に該当する場合のみ必須。該当するのに書かれていない設計は契約違反として差し戻される。

### `### 状態遷移 matrix`

- 発動条件: safety 機構・保護経路・gate・fail-closed / fail-open の挙動に触れる変更。
- 形式: 状態（normal / degraded / mismatch / missing context）× 操作（**risk-increasing / risk-reducing に分類して列挙**）× 結果（success / retryable failure / permanent failure）の表。各セルで「最終的な system state が安全側へ進むか」を確認する。
- 必須の確認: risk-reducing な操作（損切り・決済・復旧・rollback）と復旧経路が、新しい gate・validation・例外に塞がれないこと。塞がる設計は原則として欠陥である。

### `### consumer matrix`

- 発動条件: 同一の値・スコープ・例外を複数の消費側（API / UI / バッチ / 別プロセス / 別サービス）へ伝播する変更。
- 形式: consumer ごとに population / time range / truncation の扱い / empty・missing state の扱い / API・UI 上の表現。

### `### non-functional contract`

- 発動条件: DB スキーマ・クエリ・hot path・大量データ処理に触れる変更。
- 形式: 1 operation あたりの query / insert 数、boundedness（LIMIT・上限の有無）、同期 I/O の有無、想定 row 増加、parse 回数、lock / transaction 境界の budget。
- 数値の分類を必ず付ける: **既知**（出典を書く）/ **推定**（根拠を書く）/ **未測定**。未測定の項目には測定方法・fail-safe 上限・deploy 後の確認手順を書く。数値を捏造しない。

### `### deployment 手順`

- 発動条件: schema migration・permission / role・設定形式の変更。
- 形式: fresh install と upgrade の両手順、および順序制約（migration と provisioning のどちらが先か、旧バージョンとの共存期間の挙動）。

## 非該当宣言ルール

条件付きセクションが非該当の場合、セクションを書かずに `### エッジケースと決定事項` へ「<matrix 名>: 非該当（理由 1 行）」と宣言する。無言の省略は認めない。宣言の妥当性は falsifier と reviewer round 1 pass 1 の検証対象になる。

## 帰属ルール

各決定に次のいずれかを付ける。

- **（ユーザー確認済み）**: 質問・議論でユーザーが確定した決定。
- **（agent 仮決め）**: agent が独断で決めた low リスクの判断と保留論点。issue-pr-autopilot が PR の「人間に確認してほしいこと」へ転記する対象。
- **（高リスク・要人間確認）**: 高リスク未検証前提プロトコルを経て仮決めで進行した決定。reviewer round 1 pass 1 の必須反証対象であり、PR の「人間に確認してほしいこと」へ必ず転記する。

## 高リスク未検証前提プロトコル

high リスクの未検証仮定・受け入れ条件の矛盾・owner 判断が必要な論点は、単なる（agent 仮決め）で通過させない。

1. まず designer / architect 自身がコード・保存状態・運用経路（migration 履歴、設定ファイル、既存テスト）から検証を試みる。
2. 検証不能な場合 — 対話中: 構造化質問でユーザーに確定させる。
3. 検証不能な場合 — 自走中は次の 3 分岐で扱う:
   - 未確定事項が**実装方式・不可逆 migration・安全性・permission 境界のいずれかを左右する**場合: 実装へ進まず停止する。issue に質問コメントを投稿し、未解決事項として最終報告に記録して、人間の回答後に再開する。
   - **reversible かつ additive で、未確定判断を含まない stage を切り出せる**場合: スコープ判定の staged PR 分割（stage mapping）と同じ形式で、その stage だけ進行する。
   - 未確定事項が**既存状態を維持するだけで変更挙動に影響しない**場合のみ: fail-safe 側の解釈（既存状態を変更しない・自動 reconcile しない側）を仮決めとして採用し、（高リスク・要人間確認）を付けて進行する。
4. マークされた前提は reviewer round 1 pass 1 の必須反証対象になり、PR description の「人間に確認してほしいこと」へ必ず転記する。

## スケール原則

小さい変更には小さい設計を書く。必須 8 セクションは単一機能の変更なら各数行で足りる。条件付きセクションは発動条件に該当しない限り書かない。重い matrix を該当のない変更に強制しない。逆に、発動条件に該当する変更でセクションを省くことは契約違反である。
