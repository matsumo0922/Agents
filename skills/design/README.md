# design

GitHub issue や作業説明を起点に、実装前の設計を確定するためのスキルです。要件の言い換えではなく構造の決定（新規コンポーネントの居場所と責務、ファイル単位の変更マップ、状態遷移・境界の決定）を行い、issue-pr-autopilot にそのまま渡せる「## 設計」を作ります。ユーザーの指示があった場合は issue に投稿します。

## 使いどころ

- issue を issue-pr-autopilot に渡す前に、設計を確定しておきたいとき
- 「設計して」「実装方針を固めて」「## 設計を書いて」と依頼したいとき
- 複数の実装アプローチから trade-off 付きで選びたいとき

## 設計の要点

- **薄いオーケストレーター + architect サブエージェント**: コード探索と設計立案は最上位ティアの architect サブエージェントに隔離し、main agent は対象特定・質問の提示・決定の記録・issue への投稿を行う。Codex では main = Sol medium（Plan mode は high）、architect = Sol high を推奨する。確定した設計ドキュメントは main agent が読んで保持し、ユーザーとの設計議論に直接応答する（隔離するのは探索ノイズであって設計そのものではない）
- **構造判断と価値判断の分離**: どの層に置くか・どう分割するかは architect が決め、ユーザーの価値判断・リスク許容に関わる論点だけを構造化質問（1 ラウンド 2〜3 問、上限 2 ラウンド）で確定する
- **決定の帰属**: 各決定事項に「ユーザー確認済み」「agent 仮決め」を付け、仮決めは issue-pr-autopilot が「人間に確認してほしいこと」へ転記できるようにする
- **自己完結する設計**: 初見の実装者が設計とリポジトリだけで完遂できることを品質基準とし、決定は「〜とする」の形で読み手に委ねない
- **成果物の分離**: 分析過程は台帳 `/tmp/design/<slug>.md`、確定した設計は設計ドキュメント `/tmp/design/<slug>.design.md` に分け、実装エージェントには設計ドキュメントのパスだけを渡す
- **パイプラインの位置**: dig（前提への挑戦）→ design（構造の決定）→ issue-pr-autopilot（実装とレビュー）。要件が曖昧と判断した場合は design が dig をスキル内から起動して要件を確定させてから設計に入る。設計は基本ローカルに確定し、別セッション・別マシンへ渡す場合の共有ストアとして issue の「## 設計」セクションへの投稿をユーザー指示で行う

## ファイル

- `SKILL.md`: スキル本体。全体方針、台帳と設計ドキュメント、architect への指示、質問ラウンド、設計ドキュメントの形式と issue 投稿、環境別ヒントを定義します。
- `agents/openai.yaml`: UI メタ情報です。
