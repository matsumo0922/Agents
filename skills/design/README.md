# design

GitHub issue や作業説明を起点に、実装前の設計を確定して issue の「## 設計」セクションとして投稿するためのスキルです。要件の言い換えではなく構造の決定（新規コンポーネントの居場所と責務、ファイル単位の変更マップ、状態遷移・境界の決定）を行い、issue-pr-autopilot にそのまま渡せる状態を作ります。

## 使いどころ

- issue を issue-pr-autopilot に渡す前に、設計を確定しておきたいとき
- 「設計して」「実装方針を固めて」「## 設計を書いて」と依頼したいとき
- 複数の実装アプローチから trade-off 付きで選びたいとき

## 設計の要点

- **薄いオーケストレーター + architect サブエージェント**: コード読解と設計立案は最上位ティアの architect サブエージェントに隔離し、main agent は対象特定・質問の提示・決定の記録・issue への投稿だけを行う
- **構造判断と価値判断の分離**: どの層に置くか・どう分割するかは architect が決め、ユーザーの価値判断・リスク許容に関わる論点だけを構造化質問（1 ラウンド 2〜3 問、上限 2 ラウンド）で確定する
- **決定の帰属**: 各決定事項に「ユーザー確認済み」「agent 仮決め」を付け、仮決めは issue-pr-autopilot が「人間に確認してほしいこと」へ転記できるようにする
- **自己完結する設計**: 初見の実装者が設計とリポジトリだけで完遂できることを品質基準とし、決定は「〜とする」の形で読み手に委ねない
- **パイプラインの位置**: dig（前提への挑戦）→ design（構造の決定）→ issue-pr-autopilot（実装とレビュー）。設計の共有ストアは GitHub issue の「## 設計」セクション

## ファイル

- `SKILL.md`: スキル本体。全体方針、記録ファイル、architect への指示、質問ラウンド、「## 設計」の形式と投稿、環境別ヒントを定義します。
- `agents/openai.yaml`: UI メタ情報です。
